## Direct answers to the charter questions

1. **Home dashboard windows** — Revenue & nights are correct for today/yesterday/this-week/this-month across all 5 tenants. **Reservations is wrong on multi-day windows** (A1-1). Timezone: "today" is UTC, not London — a 1-hour/day BST edge (A1-6). No double-counting of revenue/nights. "Arrivals" and "Stayed" both read the *same* occupied-night map and are **identical for the today window**; "arrivals" is forward-windowed occupied-nights (not check-in counts), "stayed" is backward-windowed — a labelling/semantics subtlety, not a numeric bug for revenue/nights. Arrivals-yesterday is hard-coded zero by design.
2. **Cancellation rate / avg LOS / ADR-by-window** — all reconcile **exactly** (PASS). Only caveats: LOS outliers (A1-8) and ADR base differs from the registry (A1-5).
3. **Booked revenue date attribution** — The report SQL's 8 `raw_json` keys and the registry's strict `createdAt` **always agree** on current prod data (0 differing rows). Booked-by-date totals are correct; the two engines match to the penny. The "fuzzy" path is a latent risk only.
4. **Inventory / occupancy definition** — Report inventory = per listing-day `max(occupiedNights, calendarRateCount or fullListingCount)`, **not** lifecycle-gated by default (A1-4) and **not** `unit_count`-scaled (A1-3). Therefore: occupancy% / RevPAR are **internally consistent for the 4 single-unit tenants** but **inflated/untrustworthy for Little Feather** because of the uc=100 multi-unit listing (numerator counts up to 47 concurrent unit-nights, denominator counts 1). The signal_lab/registry occupancy (calendar_rates denominator) is **unusable** (>100% everywhere). Trustworthy occupancy today = the *reports*-path number, under the definition "occupied listing-nights ÷ (all non-removed listing-days, multi-unit collapsed to 1)". Fix A1-3 + decide A1-4 to make it defensible.
5. **Two metric engines** — They **disagree on every stay metric a user can see in two places**: stay revenue/ADR off by 1.6–20%, occupancy nonsensical in the registry. Booked revenue/bookings agree. Quantified in `a1-two-engines.ts`. The main tabs are the correct engine; signal_lab (registry) is the wrong one and is already orphaned from navigation.
